# ISO 27001 ISMS 導入文件生成系統 - 開發計畫 (Development Plan)

## 專案開發里程碑 (Milestones)

### Phase 1: 基礎環境準備與概念驗證 (POC - Proof of Concept)
* **目標**: 驗證「透過 Python 讀取 Word 範本並替換自訂變數」的可行性。
* **步驟**:
  1. [ ] 確認與設定本機 Python 開發環境 (建立虛擬環境、安裝套件 `python-docx`)。
  2. [ ] 從 `files` 中挑選一份最簡單的文件 (例如：IS-001 資訊安全政策)。
  3. [ ] 手動將該文件的關鍵資訊 (如公司名稱、日期) 修改為 `{{company_name}}`、`{{date}}` 等變數標籤。
  4. [ ] 撰寫第一支 Python 腳本 `generate_poc.py`，讀取範本、填入假資料，並另存為新檔。
  5. [ ] 檢視產出結果，確認格式未跑版。

### Phase 2: 系統核心生成邏輯 (Core Engine)
* **目標**: 建立穩健的生成引擎，支援多文件批次產出、範本鉤稽檢查與風險自動統計。
* **步驟**:
  1. [ ] 定義所有文件的層級 (四階層架構)、變數清單與資料結構 (採用 JSON 格式模擬使用者輸入)。
  2. [ ] 建立「編碼原則與自動命名模組」，確保產出檔案具備正確的 ISMS 編碼關聯。
  3. [ ] 開發 Python 模組：支援讀取單一或多個 Word 檔案進行批次替換，並加入「檢核條件提醒」與「文件鉤稽檢查」機制。
  4. [ ] 導入 Excel (`.xlsx`) 處理模組 (如 `openpyxl`)，開發「風險評鑑自動統計與分析模組」，並能將結果自動寫回至分析報告。
  5. [ ] 建立「設定檔 (Config)」機制，指定輸入範本路徑與輸出層級結構儲存路徑。

### Phase 3: 四階段導引之前端介面開發 (User Interface)
* **目標**: 依據「從零導入的四階段作業流程」設計結構化、漸進式的使用者填寫介面。
* **步驟**:
  1. [ ] **執行範本解構與問答邏輯逆推**：逐一檢視各階文件範本，盤點所有變數與需客製化條文，據此設計對應的前端問卷題目，確保問卷與範本 100% 吻合。
  2. [ ] 開發【階段一】UI：專案啟動與全景建構 (包含公司基本資料與驗證範圍動態判斷邏輯)。
  3. [ ] 開發【階段二】UI：資產盤點與風險評鑑 (包含下拉選單範本與風險運算之即時互動介面)。
  4. [ ] 開發【階段三】UI：管理程序與自訂規定檢核 (包含條文檢核防呆與「差異比對 Diff」審閱畫面)。
  5. [ ] 撰寫後端 API (使用 `Flask` 或 `FastAPI`) 來接收各階段的作答與確認結果。

### Phase 4: 系統整合、鉤稽檢查與最終封裝 (Integration & Finalization)
* **目標**: 整合前後端，完成【階段四】的體系檢查與最終文件打包匯出。
* **步驟**:
  1. [ ] 串接跨階段之前端 API 與後端 Core Engine。
  2. [ ] 實作「體系鉤稽與相依性檢查」邏輯：確保使用者無缺漏相依文件與控制項。
  3. [ ] 實作「落地導讀手冊 (ReadMe) 動態生成模組」：隨最終文件一併打包。
  4. [ ] 進行 End-to-End (端到端) 測試：模擬真實使用者自階段一至截段四填寫完整流程，確認壓縮檔打包正確。

### Phase 5: 使用者手冊與專案交付
* **目標**: 讓無程式背景的 ISO 27001 承辦人員能順利運作系統。
* **步驟**:
  1. [ ] 撰寫系統操作手冊 (`readme.md` 或 PDF 檔)。
  2. [ ] 教導如何新增或修改底層的範本文件。
  3. [ ] 提供一鍵啟動腳本 (如 `start.bat` 或 `start.sh`)，簡化啟動 Web 伺服器的流程。
